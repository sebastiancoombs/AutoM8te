### Architecture
- **realtime-session.js**: WebSocket client for OpenAI Realtime API (GA endpoint, not beta)
  - Uses `session.update` with GA event format (session.type='realtime', audio.input/output objects)
  - `semantic_vad` for turn detection (model decides when user is done speaking)
  - 12 drone tools registered as functions
  - Auto-reconnect on disconnect
  - Function call handler: parses args → HTTP to swarm manager → sends result back → triggers response.create

- **discord-voice.js**: @discordjs/voice audio capture + playback
  - DownsampleTransform: Discord 48kHz stereo → Realtime API 24kHz mono
  - UpsampleTransform: Realtime API 24kHz mono → Discord 48kHz stereo
  - Barge-in: cancels playback when user starts speaking
  - subscribeToAll() auto-subscribes to any speaker in the channel

- **drone-tools.js**: 12 tools matching swarm manager endpoints
  - All tools from voice/tools-drones.json + hover, emergency_stop from server.py
  - Each tool has name/description/parameters for Realtime API + HTTP route for execution

- **system-prompt.js**: Military copilot voice ("Drone 1, lifting off to 10 meters.")

- **index.js**: Discord bot with !join, !leave, !status, !text commands

### Key Decisions
1. **PCM 24kHz** — Realtime API GA uses `audio/pcm` format at 24000 Hz (not 16kHz like beta)
2. **semantic_vad** — Better than server_vad for drone commands (understands "take off to 20" as one utterance)
3. **Node.js** — Required by @discordjs/voice. No Python alternative for Discord voice.
4. **Direct WebSocket** — Not using Agents SDK (too much abstraction). Raw WS gives us control over audio pipeline.
5. **Barge-in** — When user speaks during playback, stop immediately. Essential for real-time control.
6. **No second bot needed** — Can share OpenClaw's token if only one connects. But recommend dedicated voice bot.

### Research: openclaw-voice-gpt-realtime
- Studied Connor Callison's OpenClaw plugin on GitHub
- It bridges Twilio ↔ OpenAI Realtime for phone calls
- Key learning: g711_ulaw passthrough (zero transcoding) — we can't do this with Discord (Opus codec)
- Their function calling pattern (DTMF, end_call, report) validated our approach
- We adapted: Twilio WebSocket → Discord @discordjs/voice, g711_ulaw → Opus↔PCM conversion
